Based on the user needs workshops and early mapping, we created some low fidelity wireframes to mock up how the core part of our tool could work.

These would be used to:

  • build out the prototype
  • give us something to refine
  • check our assumptions
  • consider technical requirements

Screenshot

Annotation view

They key things for us to consider at this stage were how the annoation view might work. We knew that there were 4 main components to this:

  • a base map to show the title plan and it’s context, we were required to use an API for Ordnance Survey base maps
  • OpenLayers to render extents over the base map, using an open GeoJSON format that will be converted from the proprietary TMF format in the existing regiser
  • text to display the entries
  • a way to define matches between extents and entries

Prototyping

We moved to higher fidelity in the prototype. The annotation page uses a two half column layout to give the same page width space to the map view, and the entry text and annotation question.

The map view uses a basemap which will use a greyscale or transparancy effect to give the extent layer a highlighted effect. We’ll explore how to do this in the next sprint. One extent will be shown at a time to avoid confusion for the user.

We’ll use radio buttons with a yes/no answer to define whether the entry describes the displayed extent or not. This will be defined for each possible extent-entry relationship so that the tool can produce a robust and accurate set of answers.

We established that we also need to include ‘other’ as answer because there are a small number of entries that desribe reationships between extents. We want to explore this further so we can give accurate answers.

In this initial design, the user will be displayed the entry text and cycle through the array of extents. We found that this was easier for the annotator to focus on than if the text changed frequently.

When all the extents have been annotated for an entry, the next button will load the next entry and cycle back through the extents. When all the entries have been annotated, the button changes to submit the title.

Annotation flow

The user need to complete an annotation task does not just require these components on a static UI in our wireframes. We also needed to consider the end to end journey and how best to design the interaction and UX so that users could effectively work through a large sequence of titles, extents and entries in a way that would give high confidence of accuracy.

We considered different options for navigating extents within a map, selecting relevant entries from a list, and cycling through entries relating to an extent.

We decided not to have a list of entries in one page, because each entry can be long and there can be lots of them. This would give cognitive overload and move the view away from the extent so it would be difficult to process.

We also decided against using checkboxes or similar to only identify relevant entries or positive matches. This carries risk of missing matches, and we know accuracy is a high priority. It also could result in lots of unknown or unannotated relationships and the admin would not have information about why. We could improve confidence that the end dataset produced was accurate by making sure that every relationship is defined as either a match or not.

We decided that extents should be processed in an order instead of selecting within a map. This means that:

  • the user can focus on understanding the text and one extent at a time
  • they don’t need to spend extra time interacting with a map, which is less accessible and more intensive
  • it reduces risk of users prioritising certain extents and gives a more standardised workflow
  • it is easier to navigate by keyboard and to complete each annotation with less work

The annotation starts from a task list page which containts the job(s) assigned to them. The job is displayed as a list of titles which forms the tasks to complete. Selecting the title starts that task of annotating it. The user continues through extents and entries for that title. When they get to the last extent and entry, the title can be submitted. Once it is submitted, the answers should be available for admin to analyse. The annotator is returned to the job page with a success notification, the status is updated to show it’s been submitted, and they may start another title.

In an earlier version, the status said ‘done’. This was changed to ‘submitted’ to match the submit action of the button which also indicates a handover to admin.

Originally we included a success interstitial page after completing a title with the option to go to the next title in the list or back to the jobs page. However, it was simpler to use a notification on the jobs page, which gives user feedback and an indication of progress through the job.
