The first task for admin is to add work to the tool for annotators to do.

In our design, work is a list of titles to be annotated. We call each list of titles a job. This is because it is a familiar term from other current workflows they may do.

To keep it simple, compiling a list of titles into a job will be done outside of the tool. We only need the title IDs to retrieve the data about them.

The admin:

  • decides which titles to include
  • puts the title IDs into one column in a csv file
  • uploads the csv file into the annotation tool

We used a file upload component to import the csv job.

Then the job needs to be assigned to an annotator. We have user profiles in our tool, so we included an assign function. For this we use a radio button list of annotator names. A job can only be assigned to one annotator. In the future, we might need to include a way to find an annotator more quickly if there are lots of annotators.

We decided to allow the admin to upload and assign one job at a time. This avoids having lots of unassigned jobs stored in the tool. It avoids risk of validation or upload failure with a batch upload of files.

There is a user need for admin to be able to assign some of the same titles to different annotators so that they can monitor the accuracy of annotation results. Uploading and assigning one job at a time enables some of the csv lists to include the same titles.

When the job is assigned, the annotator returns to their landing page and we display a success alert to confirm the action was done.
