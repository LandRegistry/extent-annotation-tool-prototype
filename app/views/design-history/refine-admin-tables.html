In the last two sprints we’ve focused on parts of the annotation user journey. For admin, we initially have a table that shows the jobs in the tool. In sprint 1, we refined an earlier version so that it did not use filters or a table with sortable headings. We revisited the jobs status views to get it ready for development in a future sprint.

Simpler sort function

We decided to simplify the sort function and just use the select input without extra styling to make a panel. This isn’t necessary and without filters doesn’t need the proximity styling to group those elements. We also moved the input into a row container above the table. This means that the table has more width if we need to add more fields in future. It also means that the action button for sorting is before the table it applies to.

Titles status

We applied the same layout to the status of titles within a job. Admin users need this view so that they can see how many tasks within a job an annotator has done, started or not started and understand how they are progressing with their work.

The titles view shows:

  • the title IDs
  • the task statuses

It does not need to include:

  • archiving or reassigning, which is done to a whole job
  • the assigned timestamp, which will be the same for all the titles in a job

We show the job ID and annotator’s name in the heading.

It also includes the simpler sort function so that users can choose which status of jobs they need to see first.

Archived jobs

The main reason for originally including filters was so that archived jobs could be not displayed by default.

Instead of using one table with filters, we decided to have separate pages with tables for active and archived jobs.

We used the same layout for each page so that the view consistently switches between them.

The archived jobs table only has one action to reassign which can be used to make the job active again.

It uses two status tags for the status which show if it was in progress or not and that it is archived. This is because users might miss that they are on the archived jobs table view. For example, the status would be incorrect if read as only ‘to do’, but ‘to do’ and ‘archived’ together are correct. This is still displayed at 400% zoom if someone is using a screen magnifier.
