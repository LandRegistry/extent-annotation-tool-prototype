After the initial design we made a few refinements to the page which provides the annotation view.

The title ID is displayed using inset text, for consistency with other pages where we provide context to the task in focus. We also add the word ‘title’ with the ID string for example ‘Title AV57267’. This is because there are a few places where we use different reference numbers, so have added labels to improve accessibility.

As we were only working with screenshots of extent maps for the first draft, we have adjusted the layout to allow more space for an OS maps container. This sets the main page width container to full width and places the map container in a govuk column half to use half of the available page space. This is based on other maps projects at HMLR which have been able to do some user testing.

We know our users will be using desktop or laptop monitors and need to compare the map with the entry text at the same time. The entry text is contained within a one third column. This is so that there is enough space for most of the text in longer entries to be displayed without losing the map container, while avoiding the length of paragraph lines becoming too long and unreadable.

We updated the matching question to make it clearer. Instead of asking the user what type of relationship there was, we ask ‘does the entry describe the selected extent?’, ‘yes’, ‘no’, or ‘other’. This gives a clear distinction and simple answer and means that all the extent-entry relationships in a title are defined.

We needed to include ‘other’ because:

  • due to data quality some extents may not have a corresponding entry
  • some entries may describe part of an extent
  • some entries may describe multiple extents, or a relationship between them

In the earlier prototype we used a back link to show the previous extent. This is because it was built using different html pages linked together. This was refined to use javascript to cycle through an array of extents. We changed ‘previous extent’ to use a secondary button as it is performing an action on the page.

‘Next extent’ uses a primary button as it is both cycing to the next extent and saving the annotator’s in progress data.

We've kept the map container on the left side of the page, followed by the entry text, radio buttons and call to action buttons on the right hand side. We wanted to have the next button on the bottom right of the page for larger screens. This will also be the submit button at the end of the annotation and is at the end of the page. The entry text, radio buttons and action buttons are grouped together in a column. We believed it could be confusing for users with screen readers if the matching input and submit button were read out before both the map and entry text had been read. Some other HMLR products use different variations of this layout but can have a lot more inputs and controls that need to feature before the map. We tried a reverse layout with the map on the right and radios below but noticed the radios could be missed. We want to test with users if this layout works well.
